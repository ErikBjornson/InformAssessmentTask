# Тестовое задание в компанию "ИнформОценка"

![Static Badge](https://img.shields.io/badge/ErikBjornson-InformAssessmentTask-InformAssessmentTask)
![GitHub top language](https://img.shields.io/github/languages/top/ErikBjornson/InformAssessmentTask)

Данный репозиторий содержит выполненное тестовое задание для устройства в компанию "ИнформОценка".

## Задача

Даны две функции ```extract_total_area(text: str)``` и ```split_num_and_text(s: str)```, расположенные в файле ```src/functions_for_testing.py```. Необходимо протестировать их функционал.

Дополнительно, в коде одной из функций допущена ошибка. Нужно найти её и исправить.

## Используемые технологии

- Python 3.11
- pytest
- allure
- flake8
- GitHub Actions

## Установка

Для локального запуска тестов сначала необходимо загрузить проект на рабочую машину. Для этого нужно:

0. Создать рабочую директорию:
```Bash
mkdir WorkingDir && cd WorkingDir
```

1. Клонировать репозиторий и перейти в папку проекта:
```Bash
git clone https://github.com/ErikBjornson/InformAssessmentTask.git
cd InformAssessmentTask
```

2. Создать и запустить виртуальное окружение Python для установки зависимостей:
```Bash
python -m venv .venv
cd .venv/Scripts
activate
cd ../..
```

3. Установить и проверить все зависимости проекта из файла ```requirements.txt```:
```Bash
pip install -r requirements.txt
pip list
```

После этого тесты готовы к запуску.

## Запуск тестов

### Локальный запуск

После клонирования репозитория, создания и запуска виртуального окружения и установки всех зависимостей проекта необходимо:

1. Перейти в корневую папку проекта:
```Bash
cd InformAssessmentTask
```

2. Выполнить команду, для запуска тестов:
```Bash
pytest -rsx
```

Ключ ```-rsx``` здесь указывается для полного вывода логов ошибок, если часть тестов окажется сломанной.

### Запуск через GitHub Actions

Для запуска тестов через GitHub Actions необходимо открыть репозиторий проекта и перейти на вкладку Actions в верхней части страницы (либо просто перейти по [ссылке](https://github.com/ErikBjornson/InformAssessmentTask/actions)).

Затем в боковом меню выбрать рабочий процесс (**workflow**) с названием ```Run flake8 and pytest tests```. Далее в правой части страницы на синем фоне появится выпадающее меню ```Run workflow```, в котором будет большая зелёная кнопка ```Run workflow```. При нажатии на неё запустится рабочий процесс и вас перебросит на страницу выполнения джоб.

![How to run tests using GitHub Actions](/assets/how_to_start_git_actions.png)

В данном проекте есть всего две джобы - ```flake8``` (линтер для проверки стиля кода) и ```tests``` (собственно, запуск тестов). Запуск тестов происходит только в том случае, если проверка линтером прошла успешно. В противном случае тесты не запустятся.

## Генерация отчётов

Для данного проекта есть возможность сгенерировать Allure-отчёт. Для этого необходимо:

0. Скачать и установить пакетный менеджер **Scoop**. Для этого необходимо запустить на локальной машине **PowerShell** НЕ от имени администратора, затем перейти на [Официальный сайт Scoop](https://scoop.sh/) и скопировать в PowerShell и запустить следующие команды:
```Bash
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
Invoke-RestMethod -Uri https://get.scoop.sh | Invoke-Expression
```

1. После установки проверить, что всё нормально установилось:
```Bash
scoop --version
```

2. Далее нужно установить сам **Allure** и после также проверить, что он нормально установился:
```Bash
scoop install allure
allure --version
```

3. Затем нужно перейти в рабочую директорию проекта, запустить виртуальное окружение и сгенерировать Allure-отчёт следующим образом:
```Bash
pytest -rsx --alluredir=allure-results --clean-alluredir
```

4. После генерации отчёта его остаётся только сформировать и запустить на локальном веб-сервере с помощью команды:
```Bash
allure serve allure-results
```

### Проводимые проверки

Для функции ```extract_total_area```:

- проверки вводимых числовых значений площади (целочисленные, дробные через точку, дробные через запятую, нулевое, отрицательное); - **найдено: 0 ошибок**
- проверки валидации единиц измерения (м2, М2, м², М², м3, сот, сот., кв.м, кв. м, м.кв, м. кв, кв.миль, га); - **найдено: 1 ошибка**
- проверки форматов ввода строки со значением площади и единицей измерения (пустая строка, без проблов, с множеством пробелов, со специальными символами, без числового значения, без единицы измерения и т.д.); - **найдено: 0 ошибок**

Для функции ```split_num_and_text```:

- проверки вводимых числовых значений цены (целочисленные, дробные через точку, дробные через запятую, нулевое, отрицательное, ценовой диапазон (в формате "от X rub")); - **найдено: 2 ошибки**
- проверки форматов ввода строки со значением цены и валюты (пустая строка, без пробелов, с множеством пробелов, со специальными символами вместо пробела, со специальными символами в валюте и т.д.); - **найдено: 4 ошибки**

### Найденные ошибки

Полный список ошибок:

![Full list of errors](/assets/all_errors.png)

1. Найдена ошибка, допущенная в коде функции ```extract_total_area```. Номер группы указан неверно, из-за этого конвертация происходила некорректно. Заменил номер группы с ```.group(2)``` на ```.group(3)```. Ошибка была в строчке:
```Python3
...
    value = float(match.group(1))
    unit = match.group(2)  <------- ЗДЕСЬ
...
```

2. Ошибка конвертации площади, где единицами измерения указаны квадратные мили (кв.миль). Регулярное выражение в функции ```extract_total_area``` распознаёт данную единицу измерения как квадратные метры (кв.м), т.к. начало обозначения квадратных миль совпадает с обозначением квадратных метров, и паттерн расценивает это как валидный объект.

Пример:
```
Ожидание: "1234.567 кв.миль" ---> None
Реальность: "1234.567 кв.миль" ---> 1234.567
```

Всё бы ничего, но нам требуется возвращать площадь либо в квадратных метрах, либо в сотках. Если и добавлять квадратные мили в расчёты, то тогда нужна корректная конвертация.

3. Ошибка при попытке отделения дробного значения цены от валюты. Эта ошибка срабатывает и при указании дробной части через запятую, и через точку. На практике у нас могут быть дробные значения цены объекта, и отделение их от валюты должно происходить корректно. В случае функции ```split_num_and_text``` мы ожидаем, что строка ```"1234.567 rub"``` станет парой ```("1234.567", "rub")```, а вместо этого получаем ```("1234", ".567 rub")```.

Пример:
```
Ожидание: "1234.567 rub" ---> ("1234.567", "rub")
Реальность: "1234.567 rub" ---> ("1234", ".567 rub")
```

4. Ошибка при указании цены без валюты. Этот случай является валидным и разделение проходит корректно, т.е. строка ```"100000"``` конвертируется в пару ```("100000", "")```. Не уверен, что это можно расценивать как ошибку, но если функция ```split_num_and_text``` нужна именно для отделения цены от валюты, и мы занимаемся оценкой стоимости жилых объектов, то валюта имеет значение, и тогда строка из примера должна конвертироваться в ```(None, None)```.

Пример:
```
Ожидание: "100000" ---> (None, None)
Реальность: "100000" ---> ("100000", "")
```

5. Ошибка очистки текста с ценой и валютой от мусорных символов. При указании некорректных символов перед ценой (например, ```"!#*^@1000000 rub"```) или вместо пробела между ценой и валютой (например, ```"1000000!#*^@rub"```) функция ```split_num_and_text``` не удаляет лишние символы и не игнорирует их. В первом примере вместо пары с разделёнными ценой и валютой функция возвращает пару ```(None, None)```. Во втором примере мусорные символы присоединяются к валюте и в итоге имеем ```("1000000", "!#*^@rub")```, что тоже неверно.

Примеры:
```
Ожидание: "!#*^@1000000 rub" ---> ("1000000", "rub")
Реальность: "!#*^@1000000 rub" ---> (None, None)
```

```
Ожидание: "1000000!#*^@rub" ---> ("1000000", "rub")
Реальность: "1000000!#*^@rub" ---> ("1000000", "!#*^@rub")
```

6. Ошибка при указании валюты с содержанием специальных символов. При написании теста я использовал такой пример: ```"1000000 !ru*bls"```. Конечно же такой валюты не существует, и функция должна была вернуть пару ```(None, None)```, однако функция ```split_num_and_text``` отделила цену от валюты и вернула пару ```("1000000", "!ru*bls")```. Здесь имеет смысл добавить в паттерн валидацию валюты (например, чтобы в валюте содержались только латинские строчные буквы).

Пример:
```
Ожидание: "1000000 !ru*bls" ---> (None, None)
Реальность: "1000000 !ru*bls" ---> ("1000000", "!ru*bls")
```
